# ADR-SEC-001: API 키 사용자 입력 + Streamlit Cloud 배포 보안

## 메타데이터
- **번호:** ADR-SEC-001
- **날짜:** 2026-07-03
- **작성자:** 한재진 (코레일유통 IT)
- **레벨:** L2
- **상태:** ✅ 승인
- **관련 ADR:** ADR-SYS-002(키 주입), ADR-SYS-001(RLS)
- **관련 프로젝트:** 규정 문서 RAG 아키텍처 비교

## 1. 맥락 (Context)
결과물을 GitHub 공개 + Streamlit Community Cloud 공개 데모로 배포한다. 공개 URL에
앱 소유자의 LLM 키가 묶이면 **아무나 접속해 소유자 비용으로 API를 소진**할 수 있고,
공개 업로드 탭은 공유 DB를 오염시킬 수 있다. 폐쇄망 원칙 조직이 클라우드를 쓰는 만큼
민감정보 유출도 차단해야 한다.

**핵심 제약 조건:**
- 공개 데모에서 소유자 키 비용 도용 차단
- 공유 Supabase 데이터 오염 방지
- 키가 저장소·로그에 남지 않아야 함

## 2. 결정 (Decision)
**채택:**
- **LLM 키(임베딩·챗·rerank)는 방문자가 사이드바에서 직접 입력** → `st.session_state`
  세션 메모리에만 보관(저장·로깅 금지). 우선순위 UI 입력 > st.secrets > .env
- Supabase 접속 정보만 소유자가 secrets로 관리
- 업로드 탭 `UPLOAD_PASSWORD` 보호
- `.gitignore`로 `.env`/`secrets.toml` 차단, `.env.example`만 커밋, 하드코딩 키 0건 스캔

## 3. 대안 검토
| 대안 | 장점 | 단점 | 탈락 이유 |
| --- | --- | --- | --- |
| 소유자 키를 secrets에 | 방문자 편의 | 비용 도용·한도 소진 | 공개 데모에 위험 |
| 데모 비공개(인증 벽 전체) | 안전 | 공유·시연 어려움 | 배포 목표와 충돌 |
| **방문자 키 입력 + 업로드 잠금** | 도용 차단·셀프 테스트 | 방문자가 키 필요 | **← 채택** |

## 4. 근거 (Rationale)
1. **비용 도용 원천 차단:** 소유자 키가 서버에 없으면 도용 불가, 방문자는 자기 키로 시험
2. **주입식 설계 정합:** Provider 생성자 키 주입(ADR-SYS-002)이라 UI 키를 그대로 사용
3. **검증:** `.py` 하드코딩 키 스캔 0건, `git check-ignore`로 `.env`/secrets 차단 확인

## 5. 결과 및 트레이드오프
**긍정적 결과:** 공개 데모 안전, 키 비저장, 업로드 오염 방지
**부정적 결과 / 위험:** 방문자가 4종 키를 준비해야 하는 진입장벽, 대상 문서(`개인정보지침.md`)
공개 여부는 별도 확인 필요(체크리스트에 명시)
**수용 기준:** 하드코딩 키 0건 + gitignore 차단 + 업로드 비밀번호 동작(충족)
