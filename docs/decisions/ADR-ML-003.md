# ADR-ML-003: 3-RAG 비교 아키텍처 + 공정 비교 원칙

## 메타데이터
- **번호:** ADR-ML-003
- **날짜:** 2026-07-03
- **작성자:** 한재진 (코레일유통 IT)
- **레벨:** L1
- **상태:** ✅ 승인
- **관련 ADR:** ADR-DATA-001, ADR-PROC-001, ADR-ML-004
- **관련 프로젝트:** 규정 문서 RAG 아키텍처 비교

## 1. 맥락 (Context)
프로젝트 목표는 Naive/Advanced/Modular 3개 RAG를 **동일 문서로 정량 비교**하는 것이다.
비교가 성립하려면 아키텍처 외의 변수를 통제해야 한다. 통제하지 않으면
"아키텍처 비교"가 아니라 "설정 비교"가 되어 결론의 신뢰가 무너진다.

## 2. 결정 (Decision)
**채택:** 3개 파이프라인 구현 + **공정 비교 원칙 고정**.
- Naive: 벡터 top-5 → 생성
- Advanced: 쿼리 재작성 → 벡터 top-20 → Cohere Rerank top-5 → 필터 → 생성
- Modular: 2단 라우팅(조번호/표/의미) → 하이브리드(BM25+벡터 RRF) → Rerank →
  조건부 인접 청크 확장 → 생성
- **고정 변수:** 동일 청크 테이블, 동일 임베딩, 동일 챗 모델, **최종 5청크 예산**,
  동일 답변 프롬프트(`prompts.py` 집중)

## 3. 대안 검토
| 대안 | 장점 | 단점 | 탈락 이유 |
| --- | --- | --- | --- |
| 파이프라인별 최적 설정 각각 | 각자 최고 성능 | 변수 뒤섞임 | 비교 불가(설정 비교로 전락) |
| 지표 없이 정성 비교만 | 간단 | 주관적 | 레포트 근거 부족 |
| **동일 조건 + 정량 지표** | 인과 해석 가능 | 통제 설계 비용 | **← 채택** |

## 4. 근거 (Rationale)
1. **변수 통제 실증:** 동일 질문에서 Naive/Advanced/Modular가 같은 청크를 검색할 때 답변 차이는 순수 아키텍처 효과로 귀속 가능
2. **차별점 관측:** 구어체 질의에서 Advanced가 제33조를 찾고 Naive는 실패, Modular 직접조회는 518토큰(의미검색 1599의 1/3)
3. **프롬프트 집중:** 답변 템플릿을 한 파일에 두어 임의 변형 차단

## 5. 결과 및 트레이드오프
**긍정적 결과:** 아키텍처 효과의 인과적 해석 가능, 비용/정확도 트레이드오프 정량화
**부정적 결과 / 위험:** "5청크 예산 고정"이 특정 파이프라인엔 불리할 수 있음(레포트에 명시)
**수용 기준:** 3파이프라인이 동일 반환 타입(RagAnswer)으로 지표 산출 가능
