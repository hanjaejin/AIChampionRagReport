# ADR-SYS-001: 벡터 저장소로 Supabase(pgvector) 채택

## 메타데이터
- **번호:** ADR-SYS-001
- **날짜:** 2026-07-03
- **작성자:** 한재진 (코레일유통 IT)
- **레벨:** L1
- **상태:** ✅ 승인
- **관련 ADR:** ADR-DATA-001, ADR-SYS-002
- **관련 프로젝트:** 규정 문서 RAG 아키텍처 비교

## 1. 맥락 (Context)
3개 RAG 파이프라인이 공유할 벡터 저장소가 필요하다. 벡터 유사도 검색 외에
BM25(하이브리드)·조번호 직접 조회·인접 청크 조회를 지원해야 한다. 학습 목적상
클라우드를 쓰되, 운영 시 폐쇄망 이식이 쉬워야 한다.

**핵심 제약 조건:**
- 무료 티어로 실습 가능
- 벡터 + 관계형 메타데이터 + 전문검색(FTS)을 한 곳에서
- 운영 전환 시 온프레미스 대체 용이

## 2. 결정 (Decision)
**채택:** Supabase(관리형 PostgreSQL + pgvector). 2테이블 구조
(`rag_documents` 1행 : `rag_chunks` N행) + RPC 3종(벡터 `match`, 하이브리드 RRF,
인접 조회) + 인덱스(HNSW, GIN, article_no/content_type). RLS 활성화.

## 3. 대안 검토
| 대안 | 장점 | 단점 | 탈락 이유 |
| --- | --- | --- | --- |
| Pinecone/Weaviate 등 전용 벡터DB | 벡터 특화 성능 | 메타/FTS 별도, 온프레 이식 어려움 | 폐쇄망 전환 부담 |
| FAISS(로컬 파일) | 무료·빠름 | 관계형 메타·동시성·FTS 부재 | 하이브리드·라우팅 구현 난이도 |
| **Supabase(pgvector)** | 벡터+관계형+FTS 통합, PostgreSQL 표준 | 관리형 의존 | **← 채택** |

## 4. 근거 (Rationale)
1. **온프레미스 1:1 이식:** pgvector는 표준 PostgreSQL 확장 → 운영 시 자체 PG로 SQL 거의 그대로 이식(ADR-SYS-002와 정합)
2. **하이브리드 지원:** tsvector(BM25) + 벡터를 RRF로 SQL 내 융합 → Modular RAG의 핵심
3. **실측:** 94청크 적재·검색·하이브리드 RPC 정상 동작 확인

## 5. 결과 및 트레이드오프
**긍정적 결과:** 단일 저장소로 벡터·메타·FTS 처리, RLS로 기본 보안
**부정적 결과 / 위험:** 무료 티어 용량/일시정지 정책, 한국어 FTS는 형태소 분석 부재(`simple`)
→ 하이브리드에서 벡터가 보완
**수용 기준:** 벡터/하이브리드/직접조회 3경로 실측 동작, RLS 활성화
