# ADR-PROC-002: TDD / Tidy First 개발 방식

- **관련 프로젝트:** 규정 문서 RAG 아키텍처 비교

## 1. 맥락 (Context)
청킹·검색·평가 로직은 외부 API 의존이 크고 실수가 조용히 잘못된 결과로 이어질 수 있다.
회귀 없이 여러 Phase를 누적 확장하려면 자동 검증 체계가 필요하다.

## 2. 결정 (Decision)
**채택:** TDD(Red→Green→Refactor) + Tidy First(구조적 변경과 행위적 변경 분리).
외부 서비스는 의존성 주입 + 가짜(fake)로 대체해 네트워크 없이 테스트. 통합은 실서비스 검증.

## 3. 대안 검토
| 대안 | 장점 | 단점 | 탈락 이유 |
| --- | --- | --- | --- |
| 사후 테스트 | 초기 속도 | 설계 결함 늦게 발견 | 회귀 취약 |
| 수동 검증만 | 도구 불필요 | 재현·확장 불가 | 8+ Phase 누적 부적합 |
| **TDD + 가짜 주입** | 회귀 방지·빠른 피드백 | 초기 작성 비용 | **← 채택** |

## 4. 근거 (Rationale)
1. **실증:** 청커에서 실측 데이터가 드러낸 호(號) 분할 누락을 2차 TDD 사이클로 즉시 보완
2. **결정성:** 토큰 계수기·임베더·스토어를 주입해 외부 없이 결정적 테스트
3. **누적 안정성:** Phase 2~8 동안 최종 79개 테스트로 회귀 방지

## 5. 결과 및 트레이드오프
**긍정적 결과:** 79개 테스트로 안전한 확장, 구조/행위 분리로 리팩터링 안전
**부정적 결과 / 위험:** 초기 테스트 작성 공수, 가짜와 실제 API 계약 불일치 가능(통합 검증으로 보완)
**수용 기준:** 각 Phase 코드가 테스트 통과 후 병합
