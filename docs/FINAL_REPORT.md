## 초록 (Executive Summary)

같은 주제(국가계약법령)의 **법률·시행령·시행규칙 3개 문서**를 함께 적재하고, **Naive /
Advanced / Modular** 3가지 RAG 아키텍처를 22문항 벤치마크로 정량 비교했다. 원래 이 프로젝트는
단일 문서(개인정보보호 지침)로 시작했으나, **여러 법령을 함께 다루는 실전 시나리오**로
문서를 교체하며 두 가지를 새로 발견했다: ① **"조번호 직접조회 = Modular 압승"이라는 이전
결론은 문서가 여러 개면 깨진다** — 같은 조번호(제26조 등)가 세 법령 모두에 존재해 문서를
구분하지 않은 직접조회가 엉뚱한 법령을 반환하기도 했다(예: "제26조 알려줘"에 시행령의
수의계약 조항만 찾고 정작 법률의 지체상금 조항을 놓침). ② **규칙 기반 라우팅(TABLE)은
원본 문서의 구조(표 존재 여부)에 암묵적으로 의존**해, 표가 없는 새 문서로 바꾸자 "서식"
질문에서 완전히 실패했다(Naive/Advanced는 정답을 찾았는데 Modular만 놓침). 그럼에도 전체
지표는 세 파이프라인 모두 0.93~0.96대로 비슷했고, Naive가 여전히 **5~6배 저렴**했다. 답변
품질(faithfulness/correctness)은 Naive·Modular가 0.955로 동률 최고였다. 이 발견들을 통해
**"RAG 아키텍처의 우열은 특정 문서·조건에서만 유효하며, 문서를 바꾸면 재검증이 필수"**
라는 교훈을 얻었다. 산출물은 실행 가능한 코드, 3탭 Streamlit 앱, 다중 문서 대응 평가
프레임워크, 공개 배포 구조, 그리고 11건의 ADR이다.

## 1. 배경

### 1.1 문제 정의
법령 문서는 장/절/조 계층 + 항/호/목이 혼재하고, 조문이 서로를 참조하며, 특히 **법률·시행령·
시행규칙처럼 여러 문서가 위임 관계로 얽혀** 있다. "수의계약을 할 수 있는 경우는?" 같은 질문에
담당자가 매번 세 문서를 오가며 뒤지는 것은 비효율적이다.

### 1.2 RAG란
RAG(검색증강생성)는 **먼저 관련 문서를 찾고(검색), 그 근거로 답을 생성**하는 방식이다.
LLM 단독은 근거 없이 그럴듯한 오답(환각)을 낼 수 있으나, RAG는 실제 조문을 근거로 제시한다.
> 비유: 사서가 질문을 듣고 관련 책을 찾아준 뒤, 그 책을 근거로 답을 정리해 주는 것.

### 1.3 조직 맥락과 제약
우리 조직은 폐쇄망이 원칙이나, 학습 목적상 클라우드(Supabase/Gemini/OpenRouter/Cohere)를
사용했다. 대신 **운영 배포 시 로컬 대체안**(8장)과 **보안 조치**(3.5·[ADR-SEC-001](decisions/ADR-SEC-001.md))를
함께 설계했다. 상세 리스크 진단은 [Phase 0 요약](progress/phase0_summary.md) 참고.

## 2. 목표
1. Naive/Advanced/Modular 3개 RAG를 **동일 조건**으로 구축·비교
2. 산출물: 실행 코드 + 성능 비교 레포트 + 사용자 매뉴얼 + 공개 데모
3. 성공 기준: **실행 가능성 · 비교 가능성(공정성) · 재현성**

## 3. 설계

### 3.1 전체 아키텍처
두 흐름으로 구성된다.
- **적재:** md 문서 → 구조 청킹 → 임베딩(Gemini) → Supabase(pgvector) 저장
- **질의:** 질문 → (파이프라인별 처리) → 근거 조문 검색 → LLM 답변 생성

### 3.2 문서 구조 분석
실사 결과 **마크다운 헤딩 불일치**(일부 조문만 `###`)를 확인했다 → 헤딩을 믿지 않고
**정규식 텍스트 패턴**으로 청킹하기로 결정. 국가계약법령 3종으로 교체한 뒤에는 **부칙의
조번호가 본문과 중복**(예: 본문 제1조 "목적" vs 부칙 제1조 "시행일")되고, **같은 조번호가
법률·시행령·시행규칙 세 문서에 모두 존재**(예: 세 문서 모두 "제1조")한다는 점을 추가로
확인했다. 후자는 평가 인프라(§7.5, 부록 A) 및 아키텍처 비교 결론(§6.3)에 직접 영향을 미쳤다.

### 3.3 청킹 전략 ([ADR-DATA-001](decisions/ADR-DATA-001.md))
- 1조=1청크(긴 조문은 항/호 단위 분할 + 조 헤더 반복)
- 1별표=1청크, **표는 자연어 캡션으로 임베딩**(마크다운 기호 노이즈 제거) — 이중 인덱싱
- `content`(답변용 원문)와 `embed_text`(검색용) 분리
- 삭제 조문 포함, "제4장" 중복은 등장 순번(`chapter_seq`)으로 구분
- 상세: [Phase 1 설계 문서](design/phase1_architecture_chunking.md)

### 3.4 데이터 모델 ([ADR-SYS-001](decisions/ADR-SYS-001.md))
`rag_documents`(문서) : `rag_chunks`(청크) 2테이블 + RPC 3종(벡터/하이브리드/인접) +
인덱스(HNSW·GIN·조번호) + RLS. 스키마: [`sql/schema.sql`](../sql/schema.sql)

### 3.5 Provider 추상화 ([ADR-SYS-002](decisions/ADR-SYS-002.md))
`ChatProvider`·`EmbeddingProvider`·`Reranker`·`VectorStore` 4개 인터페이스. 구현체만
교체하면 폐쇄망 전환 가능. API 키는 생성자 주입(UI 입력 지원).

### 3.6 3개 파이프라인 설계 비교
| 단계 | Naive | Advanced | Modular |
| --- | --- | --- | --- |
| 쿼리 전처리 | 없음 | 재작성(LLM) | 라우팅(조번호/표/의미) |
| 검색 | 벡터 top-5 | 벡터 top-20 | 하이브리드(BM25+벡터 RRF) |
| 재정렬 | 없음 | Cohere Rerank | Cohere Rerank |
| 추가 | — | 검색 후 필터 | 조건부 인접 청크 확장 |

### 3.7 평가 설계 ([ADR-ML-004](decisions/ADR-ML-004.md))
검색 지표(Recall@k/MRR/nDCG) + 답변 품질(LLM 심판, **생성과 다른 모델**로 자기채점 회피) +
비용·지연 + **오류 3단 분류**(E1 검색/E2 재정렬/E3 생성). 골드 라벨=조번호라 E1/E2는 자동 판정.

## 6. 결과

### 6.1 종합 지표 (전체 해석: [benchmark_report.md](benchmark_report.md))
| 파이프라인 | Recall@5 | MRR | nDCG@5 | Faithful | 지연(중앙) | 질의당 비용 |
| --- | --- | --- | --- | --- | --- | --- |
| **Naive** | **0.955** | 0.856 | 0.881 | 0.955 | **2.7s** | **$0.00046** |
| Advanced | 0.932 | **0.909** | **0.903** | 0.932 | 4.9s | $0.00254 |
| Modular | 0.932 | 0.879 | 0.881 | **0.955** | 4.0s | $0.00249 |

### 6.2 오류 분석
naive `21×OK+1×E1`, advanced `20×OK+1×E2+1×E3`, modular `21×OK+1×E1`. 이번엔 이상치 없이
안정적으로 실행됐다(지연 분포가 고름). 단, advanced의 E3 1건은 답변 원문 확인 결과 실제
품질 문제가 아니라 **심판 JSON 파싱 실패로 보이는 측정 아티팩트**일 가능성이 높다(§6.3, 상세는
[benchmark_report.md 발견 3](benchmark_report.md)).

### 6.3 카테고리별(핵심 — 이전 결론이 뒤집힌 지점)
- **조번호 직접조회의 이점이 사라짐:** 이전(단일 문서) 벤치마크에서 "Modular 직접조회 =
  MRR 1.00·토큰 1/3"이었으나, **법률·시행령·시행규칙이 같은 조번호를 공유**하자 Modular의
  `get_by_article`이 문서 구분 없이 3개를 모두 반환해 토큰 절감 효과가 사라지고
  (직접조회 평균 ~2,300토큰 vs 이전 ~800), MRR도 문항마다 0.33~1.00으로 들쭉날쭉해졌다.
- **q07 "제26조 알려줘"에서 Advanced가 완전히 틀린 문서만 회수(MRR 0.00).** 쿼리 재작성이
  "제26조"를 수의계약(시행령/시행규칙)으로 앵커링해 정답인 법률(지체상금)을 놓쳤다.
- **q19 "서식" 질문에서 Modular의 TABLE 라우팅이 전면 실패(Recall 0.0).** 이 코퍼스엔 표
  청크가 없어 TABLE 검색이 0건을 반환했지만, 라우팅하지 않은 Naive/Advanced는 정답을 찾았다.

### 6.4 비용·지연
Advanced/Modular는 rerank + 추가 LLM 호출로 **비용 5.5배**. 이번 실행은 레이트리밋 이상치가
없어 평균과 중앙값의 차이가 크지 않다(모두 표에 중앙값 기재).

## 7. 인사이트

1. **"이전 결론은 그 문서에서만 유효했다"가 가장 큰 교훈.** 단일 문서(개인정보지침)에서
   "조번호 직접조회 = Modular 압승"이었지만, 법률·시행령·시행규칙 3종을 함께 적재하자
   **같은 조번호가 여러 문서에 존재**해 그 이점이 사실상 사라졌다(§6.3). RAG 아키텍처의
   우열은 특정 문서·조건의 스냅샷이며, **문서를 교체·추가할 때마다 재검증이 필요**하다.
2. **규칙 기반 라우팅은 원본 문서의 암묵적 전제(예: 표 존재 여부)에 의존한다.** "서식→TABLE"
   규칙은 표가 있던 문서를 전제로 설계됐고, 표가 없는 새 문서로 바꾸자 라우팅하지 않은
   파이프라인보다 오히려 못한 결과를 냈다(q19, §6.3). 문서 교체 시 라우팅 규칙의 전제를
   먼저 재확인해야 한다.
3. **다중 문서 평가에는 "정답 문서"까지 골드에 포함해야 한다.** 조번호만으로 채점하면
   틀린 문서의 동일 번호 조문도 정답으로 오채점될 수 있다. 이번에 `doc_key_map` 기반
   문서 접두 라벨링을 도입해 이 위험을 제거했다(부록 A).
4. **그럼에도 "복잡한 RAG ≠ 좋은 RAG"는 여전히 성립.** 전체 지표는 세 파이프라인이
   0.93~0.96대로 비슷했고 Naive가 가장 저렴·빠르며 Recall도 최고였다.
5. **직접조회를 다시 강점으로 만들려면** 라우터에 법령 판별 단계를 추가하거나
   `get_by_article`에 문서 스코프를 기본 적용해야 한다(§ benchmark_report.md 추가 인사이트).
6. **상황별 선택 가이드**(쉬운 설명·택시 비유, 다중 문서 갱신 반영): [benchmark_report.md 4장](benchmark_report.md)
   - 문서가 1종뿐 → **Modular**(조번호 충돌이 없어 직접조회가 진짜 이득)
   - **문서가 여러 종류(이번처럼 3종 이상)** → **Naive** 또는 문서 스코프를 지정한 Modular
   - 명확한 질의/대량 트래픽 → **Naive**
   - 구어체·약한 임베딩 → **Advanced**(단, 조번호가 섞인 질의는 주의)
   - 실무 권장: **계단식(escalation)** — 기본은 가볍게, 실패 시에만 고급 파이프라인 발동

- **다문서 라우팅 개선(신규 최우선 과제):** `get_by_article`에 문서 스코프 기본 적용,
  질의에서 법령명 힌트 추출, 조번호 충돌 시 사용자에게 후보 문서를 되묻는 UX(§6.3·§7)
- TABLE 라우팅 규칙에 "이 코퍼스에 표 청크가 실제로 있는가" 사전 가드 추가(q19 재발 방지)
- 레이트리밋 재시도 확대, 한국어 형태소 BM25, rerank 임계치 상대컷
- 어려운 질의(다중조문·모호·오탈자)로 벤치마크 확장 → Advanced/Modular 강점 검증

### 8.3 운영 적용
사내 규정 챗봇, 접근통제·감사로그 연계, 정기 개정 반영 파이프라인.
