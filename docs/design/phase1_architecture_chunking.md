> 대상 문서: `개인정보지침.md` (코레일유통 개인정보보호 지침, 제2024-23호)

## 1. 공정 비교 원칙 (모든 설계의 전제)

3개 아키텍처를 비교하는 실험이므로, **아키텍처 외의 변수는 전부 고정**한다.
이것이 지켜지지 않으면 "아키텍처 비교"가 아니라 "설정 비교"가 되어 레포트의 설득력이 무너진다.

| 고정 항목 | 값 |
| --- | --- |
| 청킹 결과물 | 3개 파이프라인이 **동일한 청크 테이블** 공유 (표 이중 인덱싱 포함 — 인덱싱 단계 개선은 3개 모두에 동일 적용) |
| 임베딩 모델 | OpenAI text-embedding-3-small (1536차원) |
| 챗 모델 | 3개 파이프라인 동일 모델 고정 (Provider는 교체 가능하되, 한 실험 회차 안에서는 동일) |
| 최종 컨텍스트 예산 | **최종 5개 청크**로 통일 (도달 경로만 다름 — 인접 청크 확장도 이 예산 안에서 수행) |
| 답변 생성 프롬프트 | 동일 템플릿 사용 |

## 2. 3가지 RAG 아키텍처 파이프라인 비교

### 2-1. 비유로 이해하기

- **Naive RAG** = 도서관에서 사서가 질문을 듣고 **관련 있어 보이는 책 5권을 그냥 갖다 주는 것**
- **Advanced RAG** = 사서가 **질문을 먼저 다듬고**(재작성), 책 20권을 뽑아온 뒤 **정독해서 가장 좋은 5권만 추리는 것**(rerank)
- **Modular RAG** = 안내데스크가 질문 유형을 파악해 **조문 담당·서식(별표) 담당·색인 담당 중 맞는 창구로 배정**하고, 결과가 부실하면 **다시 찾아오게 시키고**(반복 검색), 찾은 페이지의 **앞뒤 페이지도 함께 복사해 주는 것**(인접 청크 확장)

### 2-2. 단계별 비교표

| 단계 | ① Naive RAG | ② Advanced RAG | ③ Modular RAG |
| --- | --- | --- | --- |
| 인덱싱(전처리) | 구조 기반 청킹 + 표 이중 인덱싱 (3개 공통) | 동일 | 동일 + **BM25용 키워드 인덱스**(PostgreSQL tsvector) 추가 |
| 쿼리 전처리 | 없음 (질문 원문 그대로) | **쿼리 재작성/확장** (LLM 1회: 구어체→규정 용어, 동의어 확장) | **라우팅**: 질문 유형 분류 → ①조번호 직접 조회 ②표(별표) 검색 ③일반 의미 검색 경로 선택 |
| 검색 | 벡터 유사도 top-5 | 벡터 유사도 top-20 (넓게 회수) | **하이브리드**: BM25 top-20 + 벡터 top-20 → RRF(Reciprocal Rank Fusion) 결합 |
| 재정렬 | 없음 | **Cohere Rerank**: 20 → 5 | **Cohere Rerank**: 20 → 5 |
| 컨텍스트 확장 | 없음 | 없음 | **조건부 인접 청크 확장** (2-4 참조, 예산 5청크 유지) |
| 반복/제어 | 없음 | 없음 | rerank 점수가 임계치 미달이면 **쿼리 변형 후 1회 재검색** |
| 컨텍스트 구성 | top-5 그대로 | rerank 상위 5 | 경로별 구성 (표 질문이면 표 청크 우선) |
| 생성 | 공통 템플릿 | 공통 템플릿 | 공통 템플릿 + 경로별 보조 지시문 |
| LLM 호출 수 | 1회 | 2회 (재작성 + 생성) | 2~4회 (라우팅 + 생성 + 재검색 시 추가) |
| 예상 지연시간 | 낮음 | 중간 (rerank API 왕복 추가) | 높음 |
| 예상 질의당 비용 | 낮음 | 중간 | 높음 |
| 강점 예상 | 단순 사실 질문 | 구어체·모호한 질문 | 조번호 명시 질문, 표(서식) 질문, 조문 간 참조 질문 |

### 2-3. 라우팅 설계 (Modular)

비용 절감을 위해 **2단 라우팅**을 채택한다:
1. **규칙 기반 1차 분류** (무료·즉시): 질문에 `제N조`·`별표 N` 패턴이 있으면 "직접 조회" 경로, "서식·표·대장" 키워드가 있으면 "표 검색" 경로
2. **LLM 분류 2차** (1차에서 미분류 시에만): 질문 유형을 LLM이 분류

> 근거: 조번호가 명시된 질문("제36조 내용 알려줘")은 벡터 검색보다 **메타데이터 정확 조회가 항상 우월**하다.
> 이 경로의 존재가 Modular의 차별성을 정량 지표로 가장 선명하게 보여줄 것으로 예상된다.

### 2-4. 인접 청크 확장 모듈 (Modular 전용) ★개정 1판 신규

규정 문서는 "전조의 규정에 따라…", "다음 각 호의 경우…"처럼 **주변 조문을 참조하는 표현**이 많다.
검색된 청크만 보면 참조 대상이 잘려 답변이 불완전해질 수 있으므로, Modular RAG에 다음 모듈을 둔다.

| 항목 | 설계 |
| --- | --- |
| 트리거 (조건부 실행) | rerank 최상위 청크가 (a) 분할 청크(`clause_range` 존재)이거나 (b) 참조 표현("전조", "제N항에 따라", "다음 각 호") 감지 시에만 실행 |
| 확장 방법 | 최상위 청크의 `chunk_index` ±1 이웃 청크를 벡터 검색 없이 **직접 조회** |
| 예산 유지 | 최종 컨텍스트는 **rerank 상위 3개 + 이웃 2개 = 총 5청크**로 재구성 (공정 비교 원칙의 5청크 예산 준수) |
| 중복 제거 | 이웃 청크가 이미 rerank 결과에 포함되어 있으면 건너뛰고 rerank 차순위로 채움. 동일 조문의 분할 청크가 겹치면 `chunk_index` 순으로 병합 정렬 |

> 조건부로 설계한 이유: 모든 질의에 무조건 확장하면 관련 없는 이웃이 컨텍스트를 오염시켜
> 오히려 faithfulness가 떨어질 수 있다. "참조가 있을 때만 이웃을 가져온다"가 안전한 기본값이다.

## 3. 청킹 전략 — 구조 기반 청킹

### 3-1. 왜 고정 길이가 아닌 구조 기반인가

- 규정 문서의 의미 단위는 **조(條)**다. "제36조가 뭐야?"라는 질문에 조문 반쪽이 잘린 청크가 검색되면 답변이 불완전해진다.
- 고정 길이(예: 500자) 청킹은 조문 경계를 무시하고 자른다 — 김밥을 재료 경계 무시하고 자르는 것과 같다.
- Phase 0 실사 결과, 이 문서는 **마크다운 헤딩이 불일치**하므로(일부 조문만 `###`), 헤딩 기호가 아닌 **정규식 텍스트 패턴**으로 경계를 인식해야 한다.

### 3-2. 경계 인식 패턴 (마크다운 헤딩 불신 원칙)

| 구조 | 인식 패턴 (개념) | 비고 |
| --- | --- | --- |
| 장 | `제N장 <제목>` | 헤딩 마크 유무 무관하게 텍스트로 인식 |
| 절 | `제N절 <제목>` | 제2장 안에만 존재 (없는 장도 있음) |
| 조 | `제N조(<제목>)` 또는 `제N조의N(<제목>)` | **1차 청크 단위** |
| 항/호/목 | `①②…` / `1. 2.` / `가. 나.` | 조 내부 분할 시에만 사용 |
| 별표 | `[별표 N]` (`##` 헤딩) | 표 청크 단위 |

### 3-3. 청크 생성 규칙

| 규칙 | 내용 | 근거 |
| --- | --- | --- |
| R1. 기본 단위 | **1조 = 1청크** | 조가 규정의 의미 단위 |
| R2. 긴 조문 분할 | 800토큰 초과 시 **항(①) 단위로 분할**, 각 분할 청크 앞에 조 헤더(`제N조(제목)`) 반복 삽입 | 임베딩 품질은 200~800토큰에서 안정적. 헤더 반복으로 분할 후에도 맥락 유지 |
| R3. 별표 | **1별표 = 1청크** (표는 분할 시 의미 파괴). 초대형 표만 행 그룹 분할 + 헤더 행 반복 | 표의 행은 헤더 없이는 해석 불가 |
| R4. 삭제 조문 | 청크로 **포함**하되 `content_type='deleted'` | "제16조 내용?"에 "삭제됨(개정 연혁)"이라 답하는 것이 침묵보다 낫다 (Phase 0 Q2 기본값) |
| R5. 제·개정 이력 | 문서 머리 연혁을 1청크(`preamble`) | "언제 개정됐어?" 질문 대응 |
| R6. 장/절 헤더 | 독립 청크로 만들지 않고 **메타데이터로만** 보유 | 헤더 단독 청크는 검색 노이즈 |
| R7. 원문/임베딩 입력 분리 ★개정 | 모든 청크는 `content`(답변 제시용 원문)와 `embed_text`(임베딩 입력)를 **분리 저장**. 일반 청크의 `embed_text` = 브레드크럼(`[제2장 > 제1절 > 제8조(…)]`) + 본문 | 임베딩에 위치 맥락 제공 + 표 이중 인덱싱(R9)의 공통 기반. 답변에는 항상 깨끗한 원문 제시 |
| R8. 전처리 정규화 | 표 셀 내 `<br>` → 공백/개행, `ㆍ` 등 특수문자 정규화, 연속 공백 정리 | 실사에서 확인된 노이즈 제거 |
| R9. 표 이중 인덱싱 ★개정 1판 신규 | `content_type='table'` 청크의 `embed_text` = 브레드크럼 + **자연어 캡션**. 캡션은 **규칙 기반 생성이 기본**: "별표 N(제목), 제M조 관련 서식, 항목: <열 헤더 나열>". LLM 요약 캡션은 옵션 플래그로 제공. `content`에는 원본 마크다운 표 유지 | 마크다운 표 기호(`\|`, `---`)는 임베딩 노이즈. 규칙 기반 캡션은 **무료·결정적·LLM 키 불필요**라 업로드 탭(임베딩 키만으로 동작)과 궁합이 맞음 |

### 3-4. 예상 청크 규모

조문 약 72개(삭제 포함) + 별표 14개 + preamble 1개 + 긴 조문 분할분 ≈ **약 100~130 청크**.
소규모이므로 검색 지연보다 **검색 품질**이 병목이며, 인덱스 전략(Phase 3)은 학습 목적으로 HNSW를 구성하되 이 규모에선 완전 탐색과 성능 차가 거의 없음을 레포트에 명시한다.

## 4. 청크 메타데이터 스키마

| 필드 | 타입 | 예시 | 용도 |
| --- | --- | --- | --- |
| `doc_id` | UUID | — | 문서 식별 (다문서 확장 대비) |
| `doc_title` | text | 개인정보보호 지침 | 출처 표기 |
| `doc_version` | text | 제2024-23호 | 개정 버전 |
| `revision_date` | date | 2024-06-27 | 문서 수준 개정일 (조별 개정일 부재 — Phase 0 Q4) |
| `chapter_seq` | int | 4, 5 | **장 등장 순번** (⚠️ "제4장" 중복 대응 — 번호 대신 순번이 유니크 키) |
| `chapter_no` / `chapter_title` | text | 제4장 / 보칙 | 표시용 |
| `section_no` / `section_title` | text, null 허용 | 제1절 / 개인정보의 처리 | 절 없는 장은 null |
| `article_no` | text | 제36조 | **평가 골드 라벨 매칭 키** |
| `article_title` | text | 권리보장의 방법 및 절차 | 표시·검색 보조 |
| `clause_range` | text, null | ①~③ | R2 분할 시 항 범위 · 인접 확장 트리거 |
| `content_type` | enum | `article` \| `table` \| `deleted` \| `preamble` | **Modular 라우팅 필터** (표 질문 → `table`만 검색) |
| `annex_no` | int, null | 3 | 별표 번호 |
| `related_articles` | text[] | {제47조} | 별표↔조 상호참조 (본문 "별표 3(제47조 관련)"에서 추출) |
| `content` | text | (원본 조문/표) | **답변 제시용 원문** (R7) |
| `embed_text` | text | (브레드크럼+본문 / 브레드크럼+표 캡션) | **임베딩 입력** (R7·R9) — 실제 벡터는 이 텍스트로 생성 |
| `table_caption` | text, null | "별표 1(개인정보파일 보유기간 책정 기준표), 제47조 관련 서식, 항목: 보유기간, 대상 개인정보 파일" | R9 캡션 원문 보존 (투명성·디버깅) |
| `chunk_index` | int | 0, 1, … | 문서 내 순서 — **인접 청크 확장(2-4)의 조회 키** |
| `token_count` | int | 512 | 비용 계산·분할 판단 |
| `embedding_version` | text | text-embedding-3-small/v1 | 모델 교체 시 재임베딩 대상 식별 |

**설계 근거 요점:**
- `article_no`를 평가의 골드 라벨 키로 삼으면, 이후 청킹 방식이 바뀌어도 **QA 평가 세트를 재사용**할 수 있다 (평가와 청킹의 결합도 분리).
- `related_articles`는 "파기 관리대장 서식 보여줘" 같은 질문에서 별표→관련 조문을 함께 제시하는 데 쓰인다.
- `content`/`embed_text` 분리는 "검색에 유리한 텍스트"와 "사람에게 보여줄 텍스트"가 다르다는 원칙의 구현이다.

## 5. 평가 오류 분석 절차 (Phase 7 평가 프레임워크에 포함) ★개정 1판 신규

아키텍처 비교의 해석이 정확하려면, 오답이 나왔을 때 **어느 단계의 책임인지** 분류해야 한다.
비유하면 배송 사고가 났을 때 "창고에 물건이 없었는지(재고), 집었는데 트럭에 안 실었는지(상차), 싣고도 잘못 배달했는지(배달)"를 구분하는 것과 같다.

| 분류 | 판정 기준 (자동 판정 가능) | 책임 소재 | 개선 방향 |
| --- | --- | --- | --- |
| **E1. 검색 한계** | 골드 조문(`article_no`)이 **1차 후보군(top-20)에 아예 없음** | 임베딩 모델·청킹·인덱스 | 임베딩 교체, 캡션/브레드크럼 개선, 하이브리드 검색 |
| **E2. 재정렬/파이프라인 실패** | 후보군엔 있었으나 **최종 5청크에서 탈락** | rerank, RRF 결합, 라우팅 | rerank 모델·임계치, 라우팅 규칙 조정 |
| **E3. 생성 실패** | 최종 컨텍스트에 골드 조문이 **있는데도 답변 오류** (faithfulness 낮음 또는 오답) | 챗 모델·프롬프트 | 프롬프트 개선, 모델 교체 |

- 골드 라벨이 `article_no` 기준이므로 E1/E2는 **LLM 없이 자동 판정**된다 (비용 0).
- 레포트에는 파이프라인별 E1/E2/E3 분포를 제시한다 — 예: "Naive의 오답 70%가 E1(검색 한계)인 반면 Advanced는 E1이 30%로 감소 → rerank가 아닌 **쿼리 재작성**이 회수율을 올렸다"는 식의 해석이 가능해진다.
- E1 비율이 3개 파이프라인에서 공통적으로 높으면 그것은 아키텍처가 아니라 **임베딩 모델의 한국어 규정 문체 한계**(Phase 0 지적)의 증거가 된다.

## 6. 온프레미스 전환 대비 추상화 경계

폐쇄망 전환 시 교체되는 것은 "구현체"이고 "인터페이스"는 유지되도록 4개 경계를 정의한다 (Phase 4에서 코드화):

| 인터페이스 | 클라우드 구현 | 폐쇄망 대체 구현 |
| --- | --- | --- |
| `ChatProvider` | OpenRouter / Gemini | vLLM(EXAONE, Qwen 등) |
| `EmbeddingProvider` | OpenAI text-embedding-3-small | BGE-M3, KURE-v1 |
| `Reranker` | Cohere Rerank | BGE-reranker-v2-m3 |
| `VectorStore` | Supabase(pgvector) | 자체 PostgreSQL + pgvector (SQL 거의 그대로) |

모든 구현체는 API 키를 **생성자 주입**으로 받는다 (UI 입력 키 정책과 폐쇄망 전환 모두의 전제 조건).

## 7. 개정 이력 및 남은 검토 관점

### 개정 1판에서 반영한 관점 (기존 "놓쳤을 수 있는 관점" 3건 채택)
1. **표 이중 인덱싱** → R9로 채택 (규칙 기반 캡션 기본, LLM 요약 옵션)
2. **인접 청크 확장** → 2-4 Modular 전용 조건부 모듈로 채택 (5청크 예산 유지)
3. **오류 분석 절차** → 5절 E1/E2/E3 분류 체계로 채택, Phase 7 평가 프레임워크 요구사항에 반영

### 이번 개정에서 새로 드러난 검토 관점
1. **규칙 기반 캡션의 품질 검증**: 캡션이 표의 내용을 잘 대표하는지 Phase 2 테스트에 "별표 캡션 생성" 케이스를 포함하고, Phase 7 평가에서 표 관련 질문의 E1 비율로 캡션 품질을 간접 측정한다.
2. **인접 확장의 중복 함정**: 분할 청크(R2)의 이웃은 같은 조문의 다른 항일 가능성이 높아, 중복 제거·병합 정렬 로직이 없으면 같은 내용이 컨텍스트에 두 번 들어간다 → 2-4 설계에 중복 제거 규칙을 명시했고, Phase 6 테스트 케이스로 검증한다.
