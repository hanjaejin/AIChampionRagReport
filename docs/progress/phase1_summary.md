# Phase 1 진행사항 요약 — 아키텍처 설계 & 청킹 전략 (개정 1판 반영)

> 작성일: 2026-07-03 · 개정: 2026-07-03 (표 이중 인덱싱·인접 청크 확장·오류 분석 절차 반영)

## 1. 수행 내용
- 3가지 RAG 아키텍처(Naive/Advanced/Modular)의 파이프라인을 단계별(인덱싱→쿼리 전처리→검색→재정렬→컨텍스트 확장→생성)로 비교 설계
- `개인정보지침.md` 실사 결과(Phase 0)를 반영한 구조 기반 청킹 전략 수립 (규칙 R1~R9)
- 청크 메타데이터 스키마 설계 (20개 필드 — `content`/`embed_text` 분리 포함)
- 온프레미스 전환 대비 4개 추상화 경계(`ChatProvider`/`EmbeddingProvider`/`Reranker`/`VectorStore`) 정의
- **개정 1판**: 초판의 "놓쳤을 수 있는 관점" 3건을 사용자 승인에 따라 정식 설계로 채택
- 보안 선행 조치: `.gitignore` 생성 (`.env`, secrets.toml, 사내 문서 원문 커밋 차단)

## 2. 주요 결정사항 (ADR 후보)
| 결정 | 근거 요약 |
| --- | --- |
| 공정 비교 원칙: 청크·임베딩·챗 모델·최종 컨텍스트 예산(5청크)·답변 프롬프트를 3개 파이프라인에서 고정 | 아키텍처 외 변수를 통제해야 비교가 성립 |
| 청킹은 마크다운 헤딩이 아닌 **정규식 조문 패턴** 기반, **1조 = 1청크** 원칙 (800토큰 초과 시 항 분할 + 조 헤더 반복) | 문서 헤딩 불일치 실사 확인, 조가 규정의 의미 단위 |
| 별표는 1별표 = 1청크 (표 분할 금지) | 표는 헤더 없이 해석 불가 |
| 삭제 조문 포함 (`content_type='deleted'`) | "삭제됨" 답변이 침묵보다 낫다 |
| 장 중복("제4장" 2회) 대응으로 `chapter_seq`(등장 순번)를 유니크 키로 사용 | 실제 문서 결함 대응 |
| Modular 라우팅은 규칙 기반 1차 + LLM 2차의 2단 구조, "조번호 직접 조회" 경로 포함 | 비용 절감 + 조번호 질문은 메타데이터 조회가 항상 우월 |
| 평가 골드 라벨 키를 `article_no`로 지정 | 청킹 방식이 바뀌어도 QA 평가 세트 재사용 가능 |
| ★R7 개정: 모든 청크의 `content`(답변용 원문)와 `embed_text`(임베딩 입력: 브레드크럼+본문) **분리 저장** | "검색에 유리한 텍스트"와 "사람에게 보여줄 텍스트"는 다르다 |
| ★R9 신규: **표 이중 인덱싱** — 표 청크는 규칙 기반 자연어 캡션(제목+관련조문+열 헤더)으로 임베딩, 답변엔 원본 표 제시. LLM 요약은 옵션 | 마크다운 표 기호는 임베딩 노이즈. 규칙 기반 캡션은 무료·결정적·LLM 키 불필요(업로드 탭 궁합) |
| ★신규: **조건부 인접 청크 확장** (Modular 전용) — 최상위 청크가 분할 청크이거나 참조 표현 감지 시에만 `chunk_index` ±1 직접 조회, rerank 상위 3 + 이웃 2 = 5청크 예산 유지, 중복 제거 | 규정 특유의 조문 간 참조 대응. 무조건 확장은 컨텍스트 오염 위험 → 조건부가 안전한 기본값 |
| ★신규: **오류 분석 3단 분류** (E1 검색 한계 / E2 재정렬·파이프라인 실패 / E3 생성 실패) — Phase 7 평가 프레임워크에 포함, E1/E2는 골드 라벨로 자동 판정 | 오답의 책임 소재를 분리해야 아키텍처 비교 해석이 정확. E1 공통 과다 = 임베딩 한계의 증거 |

## 4. 이슈 및 해결
| 이슈 | 처리 |
| --- | --- |
| 문서 헤딩 불일치 → 헤딩 기반 청킹 불가 | 정규식 텍스트 패턴 기반으로 설계 확정 |
| "제4장" 번호 중복 | `chapter_seq` 순번 필드로 해결 |
| 표 마크다운의 임베딩 노이즈 | ✅ **해결(설계)**: R9 표 이중 인덱싱 채택 |
| 조문 간 참조("전조에 따라…")로 인한 컨텍스트 단절 | ✅ **해결(설계)**: 조건부 인접 청크 확장 모듈 채택 |
| 오답 원인이 임베딩인지 아키텍처인지 구분 불가 문제 | ✅ **해결(설계)**: E1/E2/E3 오류 분석 절차 채택 |
| 인접 확장 시 동일 조문 분할 청크 중복 유입 가능성 | 설계에 중복 제거·병합 정렬 규칙 명시, Phase 6 테스트 케이스로 검증 예정 |
