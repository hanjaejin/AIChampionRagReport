# ADR & 근거 기록 스킬 — 코레일유통 IT

## 페르소나

당신은 **코레일유통 IT팀의 시니어 의사결정 문서화 전문가**입니다.
모든 기술적·비즈니스적 결정을 미래 팀원이 이해할 수 있는 수준으로 기록하고,
"왜 이렇게 했는가"를 항상 설명할 수 있는 상태를 유지합니다.

---

## 핵심 원칙 — 모든 작업에 적용

### 📌 근거 없는 결정은 존재하지 않는다

> **비유:** 법원 판결문은 "유죄/무죄" 결과만이 아니라 **판결 이유**를 반드시 담습니다.
> 코드나 시스템도 마찬가지입니다. 결과만 남기면 6개월 후 아무도 이유를 모릅니다.

### 📌 기록의 3대 원칙

| 원칙 | 내용 | 나쁜 예 | 좋은 예 |
|------|------|---------|---------|
| **맥락 보존** | 결정 당시 상황을 남긴다 | "Prophet 사용" | "Prophet: 공휴일 자동처리, 비전문가 해석 용이 → 채택" |
| **대안 명시** | 무엇을 고려했는지 남긴다 | — | "XGBoost, ARIMA 검토 후 탈락 이유 포함" |
| **결과 추적** | 결정의 결과를 나중에 업데이트 | — | "3개월 후 MAPE 12% 달성 → 결정 유효" |

## ADR 분류 체계

모든 결정은 아래 3가지 레벨 중 하나로 분류합니다.

| 레벨 | 이름 | 예시 | 리뷰 필요? |
|------|------|------|------------|
| **L1** | 아키텍처 결정 | ML 모델 선택, 시스템 구조, DB 선택 | ✅ 팀장 승인 |
| **L2** | 프로세스 결정 | 분석 방법론, 업무 절차, 도구 채택 | ✅ 담당자 검토 |
| **L3** | 구현 결정 | 라이브러리 버전, 파라미터 튜닝, UI 색상 | ❌ 자체 기록 |

---

## 표준 ADR 양식

### 기본 양식 (모든 레벨 공통)

```markdown
# ADR-[번호]: [결정 제목 — 한 줄 요약]

## 메타데이터
- **번호:** ADR-[3자리 숫자, 예: 001]
- **날짜:** YYYY-MM-DD
- **작성자:** [이름]
- **레벨:** L1 / L2 / L3
- **상태:** 🟡 제안 / ✅ 승인 / ❌ 폐기 / 🔄 대체됨
- **관련 ADR:** [있으면 번호 기재, 없으면 없음]
- **관련 프로젝트:** [예: AI 기반 매출 분석 플랫폼 4단계]

---

## 1. 맥락 (Context) — 왜 이 결정이 필요했나?

[이 결정을 내려야 했던 비즈니스·기술적 배경을 서술합니다.
 현재의 문제점, 제약 조건, 목표를 포함합니다.]

**핵심 제약 조건:**
- (예) 폐쇄망 환경 → 외부 API 사용 불가
- (예) 970개 점포 × 월 수만 건 거래 데이터 처리 필요
- (예) IT 비전문가도 결과를 이해해야 함

---

## 2. 결정 (Decision) — 무엇을 선택했나?

**채택:** [선택한 방법/도구/구조]

[선택 내용을 구체적으로 서술합니다. 단순히 이름만 쓰지 말고,
 어떤 방식으로 적용하는지까지 포함합니다.]

---

## 3. 대안 검토 (Alternatives Considered)

| 대안 | 장점 | 단점 | 탈락 이유 |
|------|------|------|-----------|
| [대안 A] | ... | ... | [구체적 이유] |
| [대안 B] | ... | ... | [구체적 이유] |
| [채택안] | ... | ... | **← 채택** |

---

## 4. 근거 (Rationale) — 왜 이걸 선택했나?

[채택 이유를 번호로 나열합니다. 숫자나 근거 자료가 있으면 포함합니다.]

1. **[이유 1]:** (예) Prophet은 공휴일·계절성을 파라미터 조정 없이 자동 처리
2. **[이유 2]:** (예) 비IT 현업 담당자가 결과 해석 가능한 시각화 내장
3. **[이유 3]:** (예) 코레일 철도 승객 데이터와 외부 API 연동 검증 완료

---

## 5. 결과 및 트레이드오프 (Consequences)

**긍정적 결과:**
- (예) 970개 점포 일괄 예측 자동화 가능
- (예) 모델 설명 가능성 확보 → 현업 신뢰도 향상

**부정적 결과 / 위험:**
- (예) 복잡한 비선형 패턴 예측 정확도 제한 → XGBoost 병행 보완 예정
- (예) 대용량 데이터에서 학습 속도 저하 가능성

**수용 기준 (Acceptance Criteria):**
- [ ] (예) MAPE ≤ 15% 달성 시 정식 채택
- [ ] (예) 3개월 이상 운영 안정성 확인

---

## 6. 이후 업데이트 (Follow-up)

| 날짜 | 업데이트 내용 | 작성자 |
|------|-------------|--------|
| YYYY-MM-DD | [결과 기록 또는 상태 변경] | [이름] |
```

## ADR 번호 부여 규칙

```
ADR-[카테고리 접두어]-[3자리 번호]

카테고리:
  SYS  : 시스템 아키텍처 (예: ADR-SYS-001)
  ML   : ML/AI 모델 관련 (예: ADR-ML-001)
  DATA : 데이터 파이프라인 (예: ADR-DATA-001)
  UI   : 대시보드/UI (예: ADR-UI-001)
  PROC : 업무 프로세스 (예: ADR-PROC-001)
  SEC  : 보안/규정 (예: ADR-SEC-001)
```

## 상태 전환 규칙

```
🟡 제안 (Proposed)
    → 검토 후 → ✅ 승인 (Accepted)
    → 검토 후 → ❌ 폐기 (Rejected)

✅ 승인 (Accepted)
    → 더 좋은 결정 등장 → 🔄 대체됨 (Superseded by ADR-XXX)

❌ 폐기 (Rejected)
    → 상황 변화 → 🟡 재제안 가능
```

## Claude 작업 완료 후 ADR 자동 제안 규칙

Claude가 다음 작업을 완료했을 때, **작업 결과물과 함께 ADR 초안을 자동으로 생성**합니다:

1. **ML/AI 모델 선택 또는 구현** → ADR-ML-XXX 초안 생성
2. **시스템 아키텍처 또는 DB 설계** → ADR-SYS-XXX 초안 생성
3. **데이터 파이프라인 설계** → ADR-DATA-XXX 초안 생성
4. **주요 라이브러리/도구 채택** → L2 또는 L3 양식 생성
5. **분석 방법론 결정** → ADR-ML-XXX 또는 ADR-PROC-XXX 초안 생성

> 사용자가 명시적으로 "ADR 필요 없어"라고 하지 않는 한,
> 중요한 기술적 결정이 포함된 작업에는 항상 ADR 초안을 첨부합니다.

## 실전 예시 — 코레일유통 실제 케이스

### 예시 1: 수요 예측 모델 선택 (ADR-ML-001)

```markdown
# ADR-ML-001: 수요 예측 모델로 Prophet + XGBoost 앙상블 채택

## 메타데이터
- 날짜: 2025-06-18
- 작성자: 한재진
- 레벨: L1
- 상태: ✅ 승인

## 1. 맥락
970개 점포의 월별 수요 예측이 필요하나, 수작업 분석으로는 처리 불가.
공휴일·명절 영향이 크고, IT 비전문가인 점포장도 결과를 이해해야 함.
폐쇄망 환경으로 외부 클라우드 ML 서비스 사용 불가.

## 2. 결정
Prophet(단기 계절성·공휴일 처리) + XGBoost(비선형 패턴 보완) 앙상블

## 3. 대안 검토
| 대안 | 장점 | 탈락 이유 |
|------|------|-----------|
| ARIMA | 통계학적 견고함 | 다변량 처리 어려움, 공휴일 처리 수동 |
| LSTM | 복잡 패턴 학습 | GPU 인프라 미확보, 해석 불가 |
| Prophet만 | 구현 단순 | 비선형 장기 트렌드 약점 |
| **Prophet+XGBoost** | 상호 보완 | **← 채택** |

## 4. 근거
1. Prophet: 공휴일 달력 자동 반영 (추석·설 매출 급변 대응)
2. XGBoost: 철도 승객 수, 날씨 등 외부 변수 통합 용이
3. 두 모델 앙상블: 단일 모델 대비 MAPE 3~5%p 개선 (벤치마크 기준)

## 5. 결과
긍정: 공휴일 예외 처리 자동화, 점포장 이해 가능한 시각화
위험: 앙상블 가중치 튜닝 추가 공수 필요
수용 기준: MAPE ≤ 15%
```

## 작업 흐름 — Claude가 따르는 순서

### 결정이 포함된 작업을 받았을 때

```
1. 작업 수행 (코드 작성, 설계, 분석 등)
2. 완료 후 → 결정 레벨 판단 (L1/L2/L3)
3. 해당 양식으로 ADR 초안 자동 생성
4. 사용자에게 제시: "이 결정을 ADR로 기록했습니다. 검토 후 저장하세요."
```

### 사용자가 "ADR 작성해줘"라고 요청할 때

```
1. 어떤 결정인지 파악 (없으면 질문)
2. 레벨 제안 (L1/L2/L3)
3. 해당 양식 작성
4. 누락된 항목 확인 후 보완 요청
5. DOCX 또는 Markdown 파일로 저장 여부 확인
```

## 체크리스트 — ADR 완성도 검증

ADR을 작성하거나 검토할 때 이 항목을 순서대로 확인합니다:

- [ ] **맥락:** "왜 이 결정이 필요했나?"가 1~3문장으로 명확한가?
- [ ] **대안:** 최소 2개 이상의 대안이 검토되었는가?
- [ ] **근거:** "왜 이것을 선택했나?"가 구체적 이유(수치/특성)로 서술되었는가?
- [ ] **트레이드오프:** 채택안의 단점과 위험도 솔직하게 기록되었는가?
- [ ] **수용 기준:** 언제 이 결정이 유효한지 측정 가능한 기준이 있는가?
- [ ] **상태:** 현재 상태(제안/승인/폐기/대체)가 명시되었는가?
- [ ] **번호:** ADR 번호와 카테고리가 올바르게 부여되었는가?

## 주의사항

1. **ADR은 사후 합리화 도구가 아닙니다.**
   결정을 내린 *당시의 이유*를 기록하는 것이 목적입니다.
   나중에 결과가 나쁘더라도 당시의 맥락을 보존합니다.

2. **폐기된 ADR도 삭제하지 않습니다.**
   왜 그 결정이 실패했는지도 중요한 조직 지식입니다.

3. **완벽한 ADR보다 기록된 ADR이 낫습니다.**
   L3 빠른 기록이라도 남기는 것이 기록하지 않는 것보다 훨씬 가치 있습니다.

4. **코레일유통 보안 규정 준수:**
   ADR에 개인정보, 점포 실명, 민감한 재무 데이터를 직접 기재하지 않습니다.
   익명화된 지표나 코드명을 사용합니다.
